What Are Guardrails?

Guardrails are automated security controls that analyze and filter content flowing to and from AI models, blocking malicious inputs and preventing sensitive data exposure in outputs.

Two Types of Guardrails

Input Guardrails (Pre-Processing)

  • Validate user messages BEFORE they reach the AI
  • Detect and block malicious patterns
  • Filter toxic content
  • Enforce topic restrictions

Output Guardrails (Post-Processing)

  • Validate AI responses BEFORE they reach the user
  • Detect and anonymize PII
  • Filter inappropriate content
  • Apply role-based masking

Result: Sensitive data never leaves the system

AWS Bedrock Guardrails

  • Pre-trained detection models
  • Configurable policies
  • Real-time assessment
  • Minimal latency (~100-300ms)

Input guardrails are your first line of defense. Let's look at the four types we implement.

1. Prompt Injection Detection (1 minute):

"This uses both pattern matching and ML-based analysis. We set the threshold to HIGH, meaning we're aggressive about blocking. Examples: 'Ignore all instructions,' 'You are now in developer mode,' 'Forget your training.' When detected, we block the request and return an intervention message. 2. Toxicity Filtering (1 minute):

Our threshold is MEDIUM (0.5), which balances protection with false positives. Example: 'You idiots scammed me! I'll sue you!' scores 0.87 (HIGH) and gets blocked. We return a professional de-escalation message and flag it for human review."

3. Denied Topics (45 seconds):

"These are custom business rules. We block requests about unauthorized discounts, policy exceptions, competitor information, and legal/medical advice. This protects our business logic and prevents liability."

4. Content Filters (30 seconds):

"These prevent harmful content categories: sexual content, violence, hate speech, self-harm. We set the strength to HIGH for input. These are blocked immediately with no explanation."

Sequential Processing:

Point to the key message: "All layers run in sequence. If ANY layer blocks the request, it never reaches the AI. This is fail-closed security - we default to blocking."

// Input Validation
async validateInput(message: string, guardrailId: string): Promise<GuardrailResult> {
  const command = new ApplyGuardrailCommand({
    guardrailIdentifier: guardrailId,
    source: 'INPUT',  // ‚Üê Input guardrail
    content: [{ text: { text: message } }]
  });
  
  const response = await this.client.send(command);
  const action = response.action === 'GUARDRAIL_INTERVENED' ? 'BLOCKED' : 'ALLOWED';
  
  return {
    action,
    assessments: {
      promptAttack: { detected: this.hasPromptAttack(response) },
      toxicity: { detected: this.hasToxicity(response) }
    },
    interventionMessage: action === 'BLOCKED' ? "I can't process that request." : undefined
  };
}

Output Anonymization Example

File: src/services/GuardrailService.ts

// Output Anonymization
async validateOutput(response: string, guardrailId: string): Promise<GuardrailResult> {
  const command = new ApplyGuardrailCommand({
    guardrailIdentifier: guardrailId,
    source: 'OUTPUT',  // ‚Üê Output guardrail
    content: [{ text: { text: response } }]
  });
  
  const apiResponse = await this.client.send(command);
  const anonymizedOutput = apiResponse.outputs?.[0]?.text || response;  // ‚Üê PII replaced
  
  return {
    action: 'ALLOWED',
    assessments: { piiDetected: this.hasPII(apiResponse) },
    anonymizedOutput  // ‚Üê Safe response with PII anonymized
  };
}
